= Jakarta WebSocket
:toc: auto

== Introduction

Jakarta WebSocket provides a standard API for building full-duplex, bidirectional communication channels over a single TCP connection in Java enterprise applications. It abstracts the complexity of the WebSocket protocol (RFC 6455), offering both annotation-based and programmatic endpoint models that integrate naturally with the broader Jakarta EE ecosystem. With support for server and client endpoints, message encoders/decoders, and lifecycle callbacks, it gives backend developers fine-grained control over persistent connections without the overhead of HTTP polling.

A WebSocket application consists of two main parts:

* A *server endpoint*, deployed in a Jakarta EE container, annotated with `@ServerEndpoint`.
* A *client endpoint*, annotated with `@ClientEndpoint`, used for server-to-server communication or testing.

== Getting Started

To use the Jakarta WebSocket API in your application, add the following dependency to your project:

[source,xml]
----
<dependency>
    <groupId>jakarta.websocket</groupId>
    <artifactId>jakarta.websocket-api</artifactId>
    <version>2.2.0</version>
    <scope>provided</scope>
</dependency>
----

=== Configuring WebSockets

No special configuration is required in `web.xml` to enable WebSocket support. The Jakarta EE container automatically detects and deploys any class annotated with `@ServerEndpoint` at application startup.

The WebSocket endpoint is accessible at the URI path declared in the annotation:

[source,java]
----
@ServerEndpoint("/chat")
public class ChatEndpoint {
    // ...
}
----

This endpoint will be reachable at `ws://host:port/app-context/chat`.

WebSocket connections begin with an HTTP upgrade handshake. The handshake URL can be secured using standard `web.xml` security constraints, just as you would for any HTTP resource. See <<Security>> for details.

=== WebSocket Usage: Server Side

Use the `@ServerEndpoint` annotation on a POJO to declare a WebSocket server endpoint. Lifecycle events are handled with method-level annotations.

[source,java]
----
import jakarta.websocket.*;
import jakarta.websocket.server.ServerEndpoint;

@ServerEndpoint("/chat")
public class ChatEndpoint {

    @OnOpen
    public void onOpen(Session session) {
        System.out.println("Connection opened: " + session.getId());
    }

    @OnMessage
    public String onMessage(String message, Session session) {
        System.out.println("Received: " + message);
        return "Echo: " + message; // returned value is sent back to the client
    }

    @OnClose
    public void onClose(Session session, CloseReason reason) {
        System.out.println("Connection closed: " + reason.getCloseCode());
    }

    @OnError
    public void onError(Session session, Throwable throwable) {
        throwable.printStackTrace();
    }
}
----

==== Sending Messages Programmatically

Instead of returning a value from `@OnMessage`, you can send messages at any time using the `Session` object:

[source,java]
----
@OnMessage
public void onMessage(String message, Session session) throws IOException {
    // Send back to the same client
    session.getBasicRemote().sendText("Echo: " + message);

    // Broadcast to all connected clients
    for (Session peer : session.getOpenSessions()) {
        if (peer.isOpen()) {
            peer.getBasicRemote().sendText("Broadcast: " + message);
        }
    }
}
----

`getBasicRemote()` sends messages synchronously. Use `getAsyncRemote()` for non-blocking sends. For binary messages, use `session.getBasicRemote().sendBinary(byteBuffer)` in the same way.

=== WebSocket Usage: Client Side

Jakarta WebSocket supports programmatic Java clients for server-to-server communication or testing.

[source,java]
----
import jakarta.websocket.*;

@ClientEndpoint
public class ChatClient {

    @OnOpen
    public void onOpen(Session session) throws IOException {
        session.getBasicRemote().sendText("Hello from Java client!");
    }

    @OnMessage
    public void onMessage(String message) {
        System.out.println("Received: " + message);
    }
}
----

To connect:

[source,java]
----
WebSocketContainer container = ContainerProvider.getWebSocketContainer();
Session session = container.connectToServer(ChatClient.class,
    URI.create("ws://localhost:8080/myapp/chat"));
----

=== Endpoint Attributes and Lifecycle Annotations

[cols="1,3", options="header"]
|===
| Annotation | Description

| `@ServerEndpoint`
| Declares a server endpoint at the given URI path.

| `@ClientEndpoint`
| Declares a client endpoint.

| `@OnOpen`
| Invoked when a new connection is established. Receives a `Session` and optionally an `EndpointConfig`.

| `@OnMessage`
| Invoked when a message is received. Can receive `String`, `ByteBuffer`, `InputStream`, or a POJO (with an encoder/decoder).

| `@OnClose`
| Invoked when the connection is closed. Receives a `Session` and a `CloseReason`.

| `@OnError`
| Invoked when an error occurs. Receives a `Session` and a `Throwable`.
|===

==== Path Parameters

URI templates support path parameters, accessible via `@PathParam`:

[source,java]
----
@ServerEndpoint("/chat/{room}")
public class RoomEndpoint {

    @OnOpen
    public void onOpen(Session session, @PathParam("room") String room) {
        System.out.println("Joined room: " + room);
    }
}
----

=== WebSocket Scopes and Session Management

Each connected client gets its own `Session` instance. The `Session` interface provides:

* `session.getId()` — unique session identifier.
* `session.getOpenSessions()` — all open sessions sharing the same endpoint.
* `session.getUserProperties()` — a `Map<String, Object>` for storing per-session state across lifecycle callbacks.
* `session.close(CloseReason)` — close the connection from the server side.

Lifecycle events can also be observed via CDI events if the container supports it. The `Session` object is always the primary handle for connection state.

To close a connection from the server with a specific reason code:

[source,java]
----
session.close(new CloseReason(CloseReason.CloseCodes.NORMAL_CLOSURE, "Session expired"));
----

== Encoders and Decoders

Encoders and decoders allow automatic serialization between Java objects and WebSocket message payloads.

=== Defining an Encoder

[source,java]
----
public class MessageEncoder implements Encoder.Text<ChatMessage> {

    @Override
    public String encode(ChatMessage message) throws EncodeException {
        return message.getUser() + ": " + message.getText();
    }

    @Override public void init(EndpointConfig config) {}
    @Override public void destroy() {}
}
----

=== Defining a Decoder

[source,java]
----
public class MessageDecoder implements Decoder.Text<ChatMessage> {

    @Override
    public ChatMessage decode(String s) throws DecodeException {
        // parse the raw string into a ChatMessage
        String[] parts = s.split(":", 2);
        return new ChatMessage(parts[0].trim(), parts[1].trim());
    }

    @Override
    public boolean willDecode(String s) {
        return s != null && s.contains(":");
    }

    @Override public void init(EndpointConfig config) {}
    @Override public void destroy() {}
}
----

=== Registering Encoders and Decoders

[source,java]
----
@ServerEndpoint(
    value = "/chat",
    encoders = { MessageEncoder.class },
    decoders = { MessageDecoder.class }
)
public class ChatEndpoint {

    @OnMessage
    public void onMessage(ChatMessage message, Session session) throws IOException, EncodeException {
        ChatMessage reply = new ChatMessage("server", "Got: " + message.getText());
        session.getBasicRemote().sendObject(reply); // uses the encoder
    }
}
----

== Security

=== Securing the Handshake Endpoint

The WebSocket handshake URL can be protected using standard `web.xml` security constraints:

[source,xml]
----
<security-constraint>
    <web-resource-collection>
        <web-resource-name>Restrict WebSocket to authenticated users</web-resource-name>
        <url-pattern>/chat</url-pattern>
    </web-resource-collection>
    <auth-constraint>
        <role-name>USER</role-name>
    </auth-constraint>
</security-constraint>
----

=== Validating the Origin

To prevent cross-site WebSocket hijacking, validate the `Origin` header during the handshake using a `ServerEndpointConfig.Configurator`:

[source,java]
----
public class OriginCheckConfigurator extends ServerEndpointConfig.Configurator {

    @Override
    public boolean checkOrigin(String originHeaderValue) {
        return "https://myapp.example.com".equals(originHeaderValue);
    }
}
----

Register it on the endpoint:

[source,java]
----
@ServerEndpoint(value = "/chat", configurator = OriginCheckConfigurator.class)
public class ChatEndpoint { ... }
----

=== Authentication Context

To access the authenticated user inside a WebSocket session, retrieve it from the `HandshakeRequest` in a custom configurator and store it in the endpoint config's user properties:

[source,java]
----
public class AuthConfigurator extends ServerEndpointConfig.Configurator {

    @Override
    public void modifyHandshake(ServerEndpointConfig config,
                                HandshakeRequest request,
                                HandshakeResponse response) {
        Principal user = request.getUserPrincipal();
        config.getUserProperties().put("user", user);
    }
}
----

Then access it inside `@OnOpen`:

[source,java]
----
@OnOpen
public void onOpen(Session session, EndpointConfig config) {
    Principal user = (Principal) config.getUserProperties().get("user");
    System.out.println("Authenticated user: " + user.getName());
}
----

== Project Information

=== Building from Source

This repository contains the source code for the specification components:

* The https://jakarta.ee/specifications/websocket/2.0/apidocs[Jakarta WebSocket API] (`/api`)
* The https://jakarta.ee/specifications/websocket/2.0/websocket-spec-2.0.html[Jakarta WebSocket specification] (`/spec`)

==== API

The Jakarta WebSocket API can be built by executing the following from the project root:

[source,bash]
----
cd api
mvn clean package
----
The API jars will be generated in `/api/client/target` and `/api/server/target`.

==== Specification

The Jakarta WebSocket specification document can be built by executing the following from the project root:

[source,bash]
----
cd spec
mvn clean package
----
The resulting zip archive containing the specification in PDF and HTML formats will be located in `/spec/target`.
